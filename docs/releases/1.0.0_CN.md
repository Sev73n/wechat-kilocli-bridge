# 版本 1.0.0

**发布日期：** 2026 年 5 月 10 日

## 概览

1.0.0 是 CLI WeChat Bridge 的第一个稳定版本，也是第一个按 npm 上架准备的版本。这个版本把项目整理成面向 Codex、Claude Code、OpenCode 和 shell 工作流的多适配器桥接工具，并强化了本地 companion 的生命周期所有权、Windows 启动行为，以及微信消息与附件发送的可靠性。

完整的 `0.9.0..1.0.0` tag 差异共涉及 61 个文件，新增 12,252 行、删除 423 行。如果不计算 release 文档，运行时代码、测试与包元数据层面的变化为 58 个文件，新增 11,912 行、删除 423 行。

### 关键统计

- **完整 tag 差异共 61 个文件**
- **新增 12,252 行，删除 423 行**
- **58 个非 release 文档文件发生变化**
- **3 条主要 companion 工作流：** Codex、Claude Code、OpenCode
- **通过 `package.json` 暴露 12 个 CLI 命令**

---

## 1. 稳定的多适配器 Companion 工作流

### OpenCode 进入主工作流

1.0.0 将 OpenCode adapter 与原生 companion 路径纳入主线。OpenCode 现在和 Codex、Claude 一样，拥有独立的 bridge 命令、companion 命令和一键启动入口。

### 新增和扩展的命令

- `wechat-bridge-opencode`
- `wechat-opencode`
- `wechat-opencode-start`
- 现有 `wechat-codex-start` 与 `wechat-claude-start` 继续沿用相同的本地 companion 模型

### 运行时行为

- 使用 OpenCode SDK，通过 HTTP 与 SSE 事件流接入
- 能跟随本地 OpenCode 会话切换，并把有意义的会话状态同步到微信
- 防止陈旧或外部 OpenCode 事件劫持当前共享会话
- 将 OpenCode companion 模型对齐到 Codex 和 Claude，使 adapter 对 bridge 生命周期的管理更一致

---

## 2. Codex Runtime 隔离与单活工作区

1.0.0 对 Codex 路径做了更严格的 runtime 隔离，引入 bridge 托管的 runtime host 和专用 remote client，避免 fallback session 与桌面端流量相互污染，并让本地工作区切换更加明确。

### 主要变化

- `wechat-codex` 改为通过 bridge 托管的 runtime host 连接，不再复用旧的本地 companion 路径
- Codex fallback session 与 Codex 桌面端流量隔离
- `wechat-codex-start` 按单活工作区切换器工作
- 在同一目录重复执行是幂等操作
- 在其他目录执行时会切换活动工作区，而不是静默创建互相竞争的会话
- 当可见端仍在但 worker 状态为 `stopped` 或 `error` 时，可以自动替换异常 bridge

### 用户影响

微信消息更不容易被送到错误的 Codex 会话，也更不容易出现重复打开可见端、或 companion 状态变化后留下陈旧 bridge 的情况。

## 3. Windows 启动与传输可靠性

这个版本包含多项面向 Windows CLI 体验的修复。

### Windows 启动修复

- Windows launcher 启动后台 bridge 时会保留代理环境变量
- 修复启动 `wechat-codex-start` 时可能闪出临时终端窗口的问题
- 固定 CLI 入口脚本行尾，避免 npm 安装后的命令 shim 与 Node shebang wrapper 在 Windows 上出现换行噪音

### Companion 断开处理

- 短暂的 companion 异常断开不会立即杀掉 bridge
- 非预期断开会保留短暂重连窗口
- 当可见端确实退出时，仍会正确关闭 companion-bound bridge

---

## 4. 消息、回复与附件可靠性

1.0.0 对微信侧的使用体验做了多处小但直接可见的修复。

### 消息投递

- 当本地 Codex 正忙时，微信文本消息会先进入队列，并在当前 turn 结束后自动投递
- 控制命令不会进入延迟队列，仍可直接处理
- 恢复 OpenCode 本地输入提示，并正确镜像真实用户消息
- 修复 Codex 启动后第一条本地消息无法同步到微信的情况

### 最终回复

- 当 Claude hook 没有提供最终回复时，会回退读取 transcript 内容
- 减少本地交互已经完成但微信侧收不到最终回复的情况

### 附件发送

- 扩展中文发文件意图识别，覆盖“给我发”“发过来”“发呀”等真实口语表达
- 将“桌面”“下载目录”等中文路径线索纳入附件判断
- 保留普通非发送语句的保护，避免误注入附件协议

## 5. npm 上架准备

1.0.0 将包整理为可发布到 npm 的形态，包名为：

```bash
npm install -g @unlinearity/cli-wechat-bridge
```

npm 包会暴露 `package.json` 中声明的 bridge 与 companion 命令，覆盖 Codex、Claude、OpenCode、shell 和版本检查入口。

### 运行要求

- Node.js `>=24.0.0`
- Bun `>=1.0.0`

### 包发布说明

- 这是 `@unlinearity/cli-wechat-bridge` 包名下第一个 npm-ready 稳定版本。
- 现有 Git clone 使用方式继续可用。
- 从 0.9.x 升级不需要做用户配置迁移。

## 重要说明

- 原有手动启动工作流仍然可用。
- `wechat-codex-start`、`wechat-claude-start`、`wechat-opencode-start` 现在应视为单活工作区启动器。
- 如果通过 npm 安装，请使用支持当前入口脚本 type stripping 要求的 Node.js 版本。
- 发布到 npm 前如果遇到认证要求，需要先运行 `npm adduser` 或 `npm login`，再执行 `npm publish --access public`。
