# 版本 1.0.3

**发布日期：** 2026 年 5 月 11 日

## 概览

1.0.3 是一个面向 Codex 与 OpenCode 实时桥接会话的可靠性补丁。它修复了两个直接影响使用体验的问题：Codex 已经把最终回答写入 session 日志但没有回传到微信，以及用户在本地 OpenCode CLI 创建新会话后，微信消息仍继续进入旧 session。

这个版本同时为 OpenCode 增加微信侧 `/new` 新建会话流程，并加强 OpenCode 回复清洗，避免 prompt 回显和同一行推理文本污染微信回复。

## 修复内容

### Codex 最终回复恢复

- 只要微信发起的 Codex turn 仍需要恢复完成状态，bridge 就会继续轮询 Codex session 日志，即使 bridge 侧 app-server 引用或连接状态已经不健康。
- 已经写入 `.codex/sessions/*.jsonl` 的 `task_complete` 现在可以恢复出缺失的 `final_reply`，避免微信侧没有回答。
- `bridge.log` 会记录 `final_reply:` 事件，并把最终回复发送失败标记为 `context=final_reply`，方便区分模型未产出和微信发送失败。

### OpenCode 新会话跟随

- OpenCode 模式下，微信现在支持 `/new` 和 `/new-session`。
- bridge 会通过 local companion proxy、local companion IPC、runtime host 和 OpenCode SDK 完整转发新建 session 请求。
- 本地 OpenCode CLI 的 session 切换现在能被更可靠地跟随，包括 `session.new`、`/new`、`tui.session.select`、`command.executed`、`session.created` 和无目录 companion `global-event` session 创建。
- OpenCode adapter 现在兼容 `sessionID`、`sessionId`、`properties.session.id`、`properties.info.id` 等常见 session 事件形态。
- 当检测到本地 OpenCode session 切换时，bridge 会清理旧 turn 状态，让后续微信消息进入新的共享 session。

### OpenCode prompt 回显与回复清洗

- OpenCode 的微信 prompt 会在 `promptAsync()` 返回前就标记为微信 turn，避免 SDK 提前发出的用户回显被误判成本地终端输入。
- 近期微信 prompt 匹配会阻止 prompt 回显进入可见输出和最终摘要。
- `The user... I should...` 这类同一行 OpenCode 推理前缀会被移除，同时保留真正回答。

## 升级说明

使用以下命令升级：

```bash
npm install -g @unlinearity/cli-wechat-bridge@latest
```

升级后，请重启正在运行的 bridge 和 companion 终端，确保它们加载 1.0.3 运行时。

OpenCode 可以在微信中发送：

```text
/new
```

也可以在本地 OpenCode CLI 里创建新 session。两种方式下，后续微信消息都应该跟随新 session。
