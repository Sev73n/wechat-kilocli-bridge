# Version 1.0.0

**Release Date:** May 10, 2026

## Overview

Version 1.0.0 is the first stable CLI WeChat Bridge release and the first release prepared for npm distribution. It turns the project into a multi-adapter bridge for Codex, Claude Code, OpenCode, and shell workflows, with stronger local companion ownership, safer Windows startup behavior, and more reliable WeChat message and attachment handling.

The full `0.9.0..1.0.0` tag delta changes 61 files with 12,252 insertions and 423 deletions. Excluding release documentation, the runtime, tests, and package changes cover 58 files with 11,912 insertions and 423 deletions.

### Key statistics

- **61 files changed** in the full tag delta
- **12,252 insertions, 423 deletions**
- **58 non-release-doc files changed**
- **3 primary companion families:** Codex, Claude Code, and OpenCode
- **12 published CLI commands** exposed through `package.json`

## 1. Stable Multi-Adapter Companion Workflow

### OpenCode support graduates into the main workflow

1.0.0 adds the OpenCode adapter and native companion path as first-class bridge targets. OpenCode now has its own bridge command, companion command, and one-click starter alongside the existing Codex and Claude workflows.

### New and expanded commands

- `wechat-bridge-opencode`
- `wechat-opencode`
- `wechat-opencode-start`
- Existing `wechat-codex-start` and `wechat-claude-start` continue to work with the same local companion model

### Runtime behavior

- Uses the OpenCode SDK with HTTP plus SSE event streams
- Follows local OpenCode session changes and mirrors meaningful session updates to WeChat
- Prevents stale or foreign OpenCode events from hijacking the active shared session
- Aligns the OpenCode companion model with Codex and Claude so the adapter owns the bridge lifecycle consistently

---

## 2. Codex Runtime Isolation and Single-Active Workspace

1.0.0 hardens the Codex path around a bridge-owned runtime host and a dedicated remote client. This prevents fallback sessions from leaking into unrelated desktop traffic and makes local workspace switching explicit.

### What changed

- `wechat-codex` now connects through a bridge-owned runtime host instead of sharing the older local companion path
- Codex fallback sessions are isolated from Codex desktop traffic
- `wechat-codex-start` works as a single-active workspace switcher
- Re-running the starter in the same directory is idempotent
- Running it from a different directory switches the active workspace instead of silently creating competing sessions
- Visible clients with `stopped` or `error` worker state can be replaced automatically

### User impact

You should see fewer cases where a WeChat message reaches the wrong Codex session, opens a duplicate visible client, or leaves a stale bridge behind after the companion state changes.

## 3. Windows Startup and Transport Reliability

This release includes several fixes aimed at making the Windows CLI experience less fragile.

### Windows launch fixes

- Preserves proxy environment variables when Windows bridge launchers spawn background processes
- Eliminates the transient terminal window that could flash when starting `wechat-codex-start`
- Pins CLI entrypoint line endings so npm-installed command shims and Node shebang wrappers remain stable on Windows

### Companion disconnect behavior

- Prevents transient companion disconnections from immediately killing the bridge
- Keeps a short reconnect window when the companion closes unexpectedly
- Continues to shut down intentionally companion-bound bridges when the visible client really exits

---

## 4. Message, Reply, and Attachment Reliability

1.0.0 improves the WeChat side of the bridge in several small but visible ways.

### Message delivery

- Defers inbound WeChat messages while local Codex is busy, then delivers them after the current turn finishes
- Keeps control commands outside the deferred queue
- Restores missing OpenCode local input prompts and mirrors real user messages correctly
- Ensures the first local message after Codex startup can sync back to WeChat

### Final replies

- Falls back to Claude transcript content when a Claude hook does not provide a final reply
- Reduces cases where a completed local interaction leaves WeChat without the expected final response

### Attachments

- Broadens Chinese send-file intent detection for phrases such as "给我发", "发过来", and short follow-up commands like "发呀"
- Treats Chinese local path hints such as "桌面" and download-folder wording as attachment clues
- Keeps ordinary non-send phrases from triggering attachment protocol injection

## 5. npm Publication Readiness

1.0.0 prepares the package for npm distribution as:

```bash
npm install -g @unlinearity/cli-wechat-bridge
```

The npm package exposes the bridge and companion commands declared in `package.json`, including Codex, Claude, OpenCode, shell, and update-check entrypoints.

### Runtime requirements

- Node.js `>=24.0.0`
- Bun `>=1.0.0`

### Important package notes

- This is the first npm-ready stable release for the package name `@unlinearity/cli-wechat-bridge`.
- Existing Git clone workflows still work.
- No user config migration is required from 0.9.x.

## Important Notes

- The existing manual workflows remain available.
- `wechat-codex-start`, `wechat-claude-start`, and `wechat-opencode-start` should now be treated as single-active workspace launchers.
- If you install from npm, use a Node.js version that supports type stripping as required by the entrypoint scripts.
- If npm authentication is required during publication, run `npm adduser` or `npm login` before `npm publish --access public`.
